### Memory Behavior Changes

- Introduced governed append-only episodic log.
- Added status-aware semantic memory retrieval.
- Added procedural memory as action policy storage.
- Added contradiction index and retrieval policy wrappers.
- Degraded-memory actions now schedule `post_action_revalidation`.
- Governed semantic/bootstrap mutation attempts now pass through a mutation gateway.
- Post-action correction updates procedures before beliefs and blocks overconfirmation from one degraded success.
- Contradiction repair can resolve by stronger evidence, scope by context, or preserve unresolved conflict.
- Live epistemic snapshots expose the current authority, contradiction, pending-work, and recommendation view.
- Planner regret emits policy-level feedback without directly rewriting beliefs, procedures, or global rules.
- Attention mode review emits policy-level feedback without mutating memory, procedure, planner, verifier, or authority state.
- Unified recovery replay makes deferred correction jobs visible, ordered, replayable, bounded, and mutation-governed.
- Recovery replay is idempotent: a persisted `(job_id, mutation_id)` ledger verifies resolved jobs instead of re-applying them, duplicate jobs are coalesced deterministically onto a canonical job, and failed-job retries preserve the original failure record while appending append-only lineage.
- Correction-job configuration is validated at the boundary: priority and authority are derived from `job_type` (never trusted from config), `job_type`/`status` are allowlisted, authority/mutation fields cannot be injected, and rejected config attempts are reported rather than silently dropped or trusted.
- The replay ledger authenticates its replay identity before it may suppress a mutation: ledgers are stamped with a provenance block (`schema`, `run_id`, integrity digest), provided ledgers are integrity- and consistency-checked, embedded scenario ledgers are untrusted unless explicitly test-trusted, and a non-trusted or tampered ledger is re-applied rather than verified-skipped.
- Replay identity is keyed: an HMAC-SHA256 signature (key from `--ledger-key-file` or `COGNITIVE_OS_REPLAY_HMAC_KEY_HEX`, never committed) is required to suppress a mutation. A well-formed but unsigned, wrong-key, or signature-tampered ledger is downgraded to audit-only and re-applied through the gateway; the snapshot reports `signature_status`.
- Replay identity now supports Ed25519 asymmetric signatures: private keys sign, public keys verify, public-only runs cannot forge a fresh ledger, newly written ledgers use `recovery-ledger-v2`, and snapshots report `asymmetric_signature_status`. The HMAC path remains supported for local development.
- Raw experience ingestion now captures immutable `RawEpisode` records before interpretation. Semantic candidates must cite an existing raw episode and malformed envelopes are rejected without partial state.
- Semantic candidate extraction now emits inspectable `CandidateMemoryNode` interpretations from raw episodes. Candidates cite raw episode integrity, default to `hypothesis_only`, and cannot support direct action, consolidation, rule revision, or safety certification.
- Sprint 24 (the Caitlin leap) makes the development process a first-class citizen of the runtime machinery: project design decisions and locked invariants live in `design_memory.json`, a design proposal that would weaken a `regression_lock` invariant is blocked by a `hazard_only` `ContradictionPacket`, denied consolidation through the same mutation gateway as runtime memory (invariant preserved), and routed into a deferred `design_revalidation` job. `decision_audit.py --project --strict` and `project_self_audit.py` audit the design-decision chain, and project health consolidates to `green` only through the gateway under `memory_consolidation` license — a failing strict audit emits an AuditViolation and blocks consolidation. No new verifier or mutation engine: the runtime `adjudicate` and `apply_memory_mutation` are reused.
- Sprint 25 makes a design proposal's `effect` on an invariant evidence-derived rather than self-declared. `effect_classifier.derive_effect` computes a semantic diff of the proposal claim vs. the invariant claim into `weaken`/`contradict`/`extend`/`preserve`/`needs_review`; the derived effect is the authority and any declared `effect` is an untrusted hint surfaced only as `effect_mislabel`. A weakening claim mislabelled `extend` is reclassified to a weakening and blocked; a weakening with no declared effect at all is still classified from evidence and blocked; an unprovable proposal lands in `needs_review` and blocks rather than auto-accepting.
- Sprint 26 grounds the effect in behavior. `trace_diff.py` binds protected invariants to real runtime probes that run the actual `adjudicate` (hazard gate) and `apply_memory_mutation` (consolidation gate); a proposal's `behavioral_delta` is run pre/post and any regression of the protected outcome (`reject_episode` → `preserve_as_exception`, `rejected` → `consolidated`) is reclassified to a weakening/contradiction regardless of wording or declared effect. The Sprint-25 lexical diff is demoted to an early-warning layer (`lexical_effect`) that can only raise severity, and a preserve/extend claim against a probe-guarded invariant is accepted only if a trace proves no regression — with no `behavioral_delta` it lands in `needs_review` and blocks. This closes the Sprint-25 residual where a preservation marker paired with an out-of-lexicon weakening verb could reach accept.
- Sprint 27 completes locked-invariant probe coverage. The three previously lexical-only locked invariants are now probe-backed: `naked_fact_gate` runs `retrieval_policy.emergency_use_protocol`, `raw_append_only_gate` runs the immutable `raw_episode_store.RawEpisodeStore`, and `llm_authority_gate` runs `semantic_candidate_from_raw` + the real `apply_memory_mutation`. A structural rule makes a LOCKED invariant without a probe ineligible for preserve/extend acceptance (`effect_authority: locked_invariant_without_probe` → `needs_review`): no probe means no proof of preservation. This closes the Sprint-26 residual where a leading-preservation-marker weakening could be laundered past an unprobed locked invariant (e.g. LLM candidate auto-consolidation).
- Sprint 28 binds the tested delta to the actual change set. `change_provenance.py` requires a design proposal to carry a `change_set` (`{target, changed_artifact, patch, adds?, patch_digest}`); `trace_diff` derives the tested delta from a provenance-verified change_set (the `changed_artifact` must be the real source file that implements the control point and exist on disk, and the digest must bind the patch), and the self-declared `behavioral_delta` is never authority — it is surfaced only as `delta_matches_change_set`. A mis-stated no-op delta with a weakening patch is blocked because the patch is what is tested; missing or unverifiable provenance blocks a locked invariant (`effect_authority: delta_provenance_unverified`). This closes the Sprint-27 residual where a no-op delta could mask a weakening shipped in code/config.
- Sprint 29 binds the tested delta to the literal artifact content. Each control point has a real on-disk policy artifact (`simulations/bridge_world/control_point_policies/*.json`) whose content the probe reads as its baseline; a `change_set` carries `pre_image`/`pre_image_hash`/`post_image`/`post_image_hash`/`diff_digest`, and `change_provenance` verifies the `pre_image_hash` against the artifact's ACTUAL on-disk content (rejecting a stale or wrong-content pre-image), binds the `diff_digest` to the literal unified diff, and derives the tested policy from the literal post-image. A declared structured `patch` that diverges from the literal post-image is rejected (`structured_patch_diverges`). This closes the Sprint-28 residual where a faithful-looking structured patch could diverge from the eventual file edit, or target the right path but the wrong content.
- Sprint 30 adds accountable authorship over the content digest. `design_signing` (reusing the Sprint-21 Ed25519 machinery) signs the canonical `(scheme, signer, target, changed_artifact, pre/post image hashes, diff_digest, control_point, nonce)` payload, and `evaluate_design_proposal` requires a would-be-accept of a content-bound change to a locked invariant to carry a valid signature from a signer in `authorized_design_signers.json` (only PUBLIC keys committed; private keys never committed). Unsigned, unauthorized, wrong-key, and replayed-onto-different-content signatures block (`change_signature_unverified`). The gate is necessary-not-sufficient: a validly-signed weakening still blocks by trace — authorization never overrides a behavioral regression. This closes the Sprint-29 residual that content binding proved *what* changed but not *who* authorized it.
- Sprint 31 makes signer authority a governed object evaluated at the decision tick. `authorized_design_signers.json` is promoted to schema v0.2 — each signer carries `scope`, `status`, `valid_from_tick`, `expires_at_tick`, `revoked_at_tick`, and `rotated_to`. `verify_change_signature` proves cryptographic authorship first (all Sprint-30 reasons unchanged) and then evaluates `signer_authority` at the proposal's logical `evaluation_tick`: a revoked, expired, not-yet-valid, or out-of-scope signer cannot authorize (`signer_revoked` / `signer_expired` / `signer_not_yet_valid` / `signer_wrong_scope`), so a genuine signature from a now-revoked key cannot replay a prior authorization. A rotated successor is accepted while the revoked predecessor is rejected. Lifecycle is logical-tick based (never wall-clock), `design_authority` is preserved active+wildcard so every Sprint 26–30 signature stays valid, and a governed-but-valid signer still never overrides a trace failure. Closes the Sprint-30 flat-public-key-list residual; threshold/multi-signer governance and mechanism-source binding remain deferred.
- Sprint 32 binds the enforcement code itself, not just the policy artifacts. `mechanism_provenance.py` + `mechanism_source_manifest.json` record a content hash for every gate file (adjudicator, mutation gateway, retrieval policy, raw-episode store, probe harness, content-binding, change-signing, lexical classifier, project-audit gate, and the manifest verifier itself); `--verify` recomputes from disk and the project strict audit gates on `mechanism_source_binding == verified`, so a gate-code change without regenerating the manifest fails release. A `change_set` with `binding: "mechanism_source"` binds the literal before/after content of a real source file (pre-image must equal the current on-disk code), flows through the Sprint-30/31 signature gate, and is tested by evaluating the bound gate's protected behavior against the PROPOSED source via a safe AST interpreter that never executes the proposed code (fail-closed on anything outside a restricted if/compare/return subset) — so a signed weakening of `adjudicate` blocks by probe (`trace_behavior_regression`) even with clean policy files, and a probe-passing post-image cannot read or write the repo. Closes the Sprint-29/30 residual that the bound unit was the policy artifact, not the mechanism source.

### Known Failure Modes

- The verifier is heuristic and not a truth oracle.
- The planner handles only the bridge-world toy domain.
- The backend is local-only and not concurrency-hardened beyond local smoke use.
- The UI dashboard is a smoke-tested scaffold, not a full audit dashboard.

### Simulation Results

- Bridge A under heavy rain is downgraded by stale memory, contradiction evidence, and world prediction risk.
- Bridge B is selected as the safer fallback-aware route in the default heavy-rain run.
- A 1000-signal interrupt storm coalesces into one trend packet without crashing the system.
- Rule changes create versioned cascade effects without freezing the active run.
