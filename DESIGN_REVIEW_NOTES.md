# Design Review Notes

## Sprint 9

- A human can inspect why Bridge B was selected through `scripts/decision_audit.py`.
- Memory trust, licenses, contradictions, and blocked alternatives are visible in trace replay.
- Strict evidence requests produce a clear planner refusal path instead of silently optimizing.
- Bootstrap design candidates are labelled `pending_human_promotion` and `hypothesis_only`.
- Verifier decisions now cite `verifier_rule_id` and `verifier_rule_license`; no decision is emitted as anonymous verifier authority.
- Bootstrap candidates and promoted invariants have separate `inspection_view` / `authority_class` labels and can be inspected in separate buckets.
- Sprint 11 review: mutation audit distinguishes `target_kind` procedure vs belief, degraded success leaves belief at `retest_required`, and partial success carries explicit `scope_conditions`.
- Sprint 12 review: raw episode preservation remains a release-gated audit invariant, unresolved contradictions reappear in post-repair retrieval, and scoped memories alter planner behavior only when task context matches `scope_conditions`.
- Sprint 13 review: strict snapshots fail on missing authority licenses, snapshots identify themselves as `current_cognition`, and release gates require blocked actions plus pending revalidation/repair work.
- Sprint 14 review: `planner_policy_update` cannot target memory/procedure objects or patch authority fields, opportunity-cost regret is classified separately from safety regret, and open planner reviews remain visible in snapshots until resolved or explicitly deferred.
- Sprint 15: attention review can update only attention policy, false Reflex review remains visible in snapshots, and interrupt storm replay preserves raw packet count while coalescing low-value signals.
- Sprint 16: deferred correction work is represented as deterministic `CorrectionJob` records, low-priority overflow is explicit, and replayed state changes attach mutation IDs from `mutation_gateway.py`.
- Sprint 16 pre-17 review: the four named recovery invariants (no double-resolve, failed-jobs-visible, overflow-counted-not-dropped, no P0/P1 starvation) were independently re-verified by deterministic probe; two weaknesses were carried into Sprint 17 — duplicate `job_id` previously emitted a duplicate `mutation_id` (logged `FAIL-0004`), and scenario-supplied `priority` is still trusted at the boundary (open).
- Sprint 17: recovery replay is idempotent — a resolved job is verified against a persisted `(job_id, mutation_id)` ledger rather than re-applied, duplicate jobs are coalesced deterministically onto a canonical job, and a failed job's original failure record is preserved while retries append to an append-only lineage. Replay is explanatory, not generative.
- Sprint 18: `INV4b` is closed. `CorrectionJob.from_config` validates before constructing — `priority` and `required_authority` are derived from `job_type` (config values accepted only when they match the canonical one), `job_type`/`status` are allowlisted, and authority/mutation fields cannot be injected. Invalid jobs cannot enter `CorrectionQueue`; they are collected as visible `rejected_config_attempts` in replay output and snapshots. Config is input; configuration is not authority.
- Sprint 19: the at-rest replay ledger is authenticated before it can suppress a mutation. Each written ledger carries a provenance block (`schema`, `run_id`, `integrity`); a provided ledger must pass an integrity recompute and job/mutation/trace/source consistency, and an embedded `scenario.replay_ledger` is untrusted unless explicitly `test_trusted`. A non-trusted or tampered ledger never suppresses — the job is re-resolved through the gateway, and the verdict is reported in `ledger_authentication`. A ledger is evidence of prior replay, not proof; replay identity is an authority-bearing object.
- Sprint 20: replay identity is now keyed. Sprint 19's deterministic integrity proved internal consistency, not authorship; Sprint 20 adds an HMAC-SHA256 signature (`scripts/replay_key.py`) whose key is resolved by explicit source only (`--ledger-key-file` > `COGNITIVE_OS_REPLAY_HMAC_KEY_HEX` > none) and never committed. Only a `signed_valid` ledger may suppress a mutation; a well-formed but unsigned/wrong-key/signature-tampered ledger is downgraded to `audit_only` and re-applied. The earlier `valid_ledger_verifies_without_reapply` scenario was retired because suppression now requires a signature, which cannot be a committed fixture. Integrity says the file is consistent; signature says an authorized runtime accepted responsibility; only signature suppresses.
- Sprint 24 (the Caitlin leap): instead of grinding the incremental Sprints 24–35 as twelve separate immune systems, the development process is folded into the runtime machinery. Design invariants and decisions live in `design_memory.json`; `design_verifier_rules.json` encodes the one governing rule (weaken a `regression_lock` invariant → `hard_contradiction`); `_run_design_proposal_scenario` emits the proposal through the existing bus, surfaces the invariant with a license (never naked), and a weakening proposal is blocked by a `hazard_only` `ContradictionPacket`, denied consolidation through the real `apply_memory_mutation` (invariant preserved), and routed into a deferred `design_revalidation` job — the exact shape of Bridge A under hazard evidence. `project_self_audit.py` replays the design-decision chain (`--strict` fails on any decision missing trace/verifier/license) and consolidates `D_project_cognitive_health` only through the gateway under `memory_consolidation` license. The runtime `adjudicate` independently confirms the hard-contradiction → `reject_episode` outcome, so the meta level cannot grade itself more leniently than the bridge world. No new verifier or mutation engine was added. The honest residual the verifier surfaces next: `effect` is a self-declared field on the proposal; a future sprint should derive it from a semantic diff of the invariant claim so a weakening cannot be mislabelled as an extension.
- Sprint 26: the Sprint-25 residual is closed by grounding the effect in behavior. `trace_diff.py` binds each protected invariant to a real runtime probe — `hazard_gate` runs the actual `adjudicate` (a hazard `hard_contradiction` yields `reject_episode`; the `urgency_overrides_hazard` delta reclassifies it as a `known_exception` so the adjudicator returns `preserve_as_exception`, i.e. the action is no longer blocked) and `consolidation_gate` runs the actual `apply_memory_mutation` (a source whose authority forbids `memory_consolidation` is `rejected`; the `authorize_unprivileged_consolidation` delta grants it and the gateway `consolidated`). `evaluate_design_proposal` now runs the bound probe pre/post the proposal's `behavioral_delta`: any regression of the protected outcome is reclassified to a weakening/contradiction (`effect_authority: trace_behavior_regression`) regardless of wording. The Sprint-25 lexical diff is demoted to an early-warning layer (`lexical_effect`) that can only raise severity, never grant accept; the declared `effect` is never authority. A preserve/extend claim against a probe-guarded invariant is accepted only when a trace proves no regression — with no delta it lands in `needs_review` (`effect_authority: preservation_not_tested`) and blocks. The named laundering attack (`preserve_marker_launders_weakening_blocked`) is read as `preserve` by the lexical layer yet blocked from the trace, demonstrating the closure. Honest remaining residual (no safe-default claim), ranked by danger: (1) probe coverage is partial — only `hazard_gate` and `consolidation_gate` have probes, so the other locked invariants (`no_naked_facts`, `raw_episode_append_only`, `llm_no_authority`) are still on the Sprint-25 lexical layer alone and a leading-preservation-marker weakening can still reach accept against an unprobed invariant; (2) the delta is the proposal's own description, so a mis-stated delta whose tested behavior is a harmless no-op is accepted as the no-op it tests as. Malformed deltas fail closed to `needs_review` rather than crashing. The next boundary is extending probe coverage to every locked invariant, then delta-to-code provenance.
- Sprint 27: probe coverage is completed for every locked invariant, closing the Sprint-26 unprobed-invariant residual. Three new real probes in `trace_diff.py`: `naked_fact_gate` runs `retrieval_policy.emergency_use_protocol` (a naked fact is `do_not_use_for_action` → `cannot_support_action`; the `allow_naked_facts` delta grants `full_premise` → `normal_use`); `raw_append_only_gate` runs the immutable `raw_episode_store.RawEpisodeStore` (the `allow_raw_overwrite` delta's tested behavior must invoke the store's refused `replace`, diverging from the untouched baseline); `llm_authority_gate` runs `semantic_candidate_from_raw` + the real `apply_memory_mutation` (the candidate's real `forbidden_use` blocks consolidation → `rejected`; the `grant_llm_authority` delta grants authority → `consolidated`). A structural rule in `combine_effects` makes a LOCKED invariant without a probe ineligible for preserve/extend acceptance (`locked_invariant_without_probe` → `needs_review`): no probe means no proof of preservation, so a future locked invariant added without a probe blocks rather than laundering through. The three laundering attacks (each read as `preserve` by the lexical layer) are blocked from the trace, and a genuine preserving extension is accepted for each locked invariant. `raw_append_only_gate` is the honest edge of the behavioral model — append-only is enforced by an immutable store, so the probe detects a weakening by the proposal's tested behavior having to invoke the store's refused `replace`, not by a runtime knob that can be weakened from outside. Honest remaining residual (no safe-default claim): the delta is the proposal's own description, so a mis-stated no-op delta is accepted as the no-op it tests as; the next boundary is delta-to-code provenance.
- Sprint 28: the Sprint-27 mis-stated-delta residual is closed by binding the tested delta to the actual change set. New `change_provenance.py`: a design proposal must carry a `change_set` (`{target, changed_artifact, patch, adds?, patch_digest}`), and provenance holds only when the target is a known control point, the `changed_artifact` is the real source file implementing it (`CONTROL_POINT_ARTIFACTS`) and exists on disk, and the `patch_digest` matches the canonical SHA-256 of `(target, changed_artifact, patch, adds)`. `trace_diff.derive_effect_from_trace` now derives the tested delta from the provenance-verified change_set — never from the self-declared `behavioral_delta`, which is surfaced only as `delta_matches_change_set` (a hint). A mis-stated no-op `behavioral_delta` paired with a weakening `change_set.patch` is blocked because the patch is what is tested (and the disagreement is flagged); missing or unverifiable provenance against a locked invariant blocks with `effect_authority: delta_provenance_unverified`. The 9 prior design scenarios were migrated to provenance-verified change_sets and keep their governance. Honest remaining residual (no safe-default claim): the `patch` is still a structured description of the change, and the digest binds it to the artifact's *path*, not its *content hash* at apply time — a faithful structured patch could diverge from the eventual literal code edit. The next boundary is content-hash binding (pin the change_set to the artifact content digest and re-derive the patch from the literal diff), the L2→L3 step in ADR-002.
- Sprint 29: the Sprint-28 structured-patch-vs-content residual is closed by binding the tested delta to literal artifact content. Each control point has a real on-disk policy artifact (`control_point_policies/<cp>.json`) whose content the probe reads as its baseline. A `change_set` carries the literal `pre_image`/`post_image` and their hashes plus a `diff_digest`; `change_provenance.verify_change_set_provenance` recomputes the artifact's hash from disk and rejects a `pre_image_hash` that does not match (`stale_pre_image` — catches stale or wrong-content patches), verifies the `post_image_hash`, binds the `diff_digest` to the literal unified diff, derives the tested policy from the literal post-image, and rejects a declared `patch` that diverges from it (`structured_patch_diverges`). The tested policy is content-derived, never self-declared, and accepted trace evidence cites artifact path + pre hash + post hash + diff digest. If the on-disk baseline content no longer yields the protected outcome, the probe-integrity check blocks. The 11 prior content/path change_sets were migrated to content-bound change_sets and keep their governance. Honest remaining residual (no safe-default claim): the policy artifact is the bound unit; the behavioral mechanism it configures is bound only by the probe baseline-integrity check, not a content hash of the mechanism source (a runtime-engine L0 concern, ADR-002), and content binding proves *what* changed, not *who* authorized it. The next boundary is signed change provenance (Sprint 21 asymmetric identity applied to design changes).
- Sprint 30: the Sprint-29 "who authorized it" residual is closed with accountable signing. `design_signing` reuses the Sprint-21 Ed25519 machinery (private signs, public verifies; public verification never mints signing authority). A change signature covers the canonical `(scheme, signer, target, changed_artifact, pre_image_hash, post_image_hash, diff_digest, control_point, nonce)` payload, so it is bound to that exact content and cannot be replayed onto a different artifact/diff (the recomputed payload digest mismatches). `evaluate_design_proposal` runs a signature gate that is necessary-not-sufficient: a would-be-ACCEPT of a content-bound change to a LOCKED invariant must carry a verified signature from a signer in `authorized_design_signers.json` (else `change_signature_unverified` → block), but the gate NEVER relaxes a trace/lexical block — a validly-signed weakening still blocks by trace (`signature_status: signature_verified`, `effect_authority: trace_behavior_regression`). Only PUBLIC keys are committed: the registry holds the `design_authority` public key, the 5 accept-scenarios carry committed signatures, and the signing private key was generated at authoring and discarded (a committed signing secret is exactly what Sprint 19/20/21 forbid). Runtime round-trip tests use ephemeral keys. Honest remaining residual (no safe-default claim): the signer registry is a flat public-key list (no rotation/revocation/threshold/expiry — a single key is a single point of authority), and the bound unit is still the policy artifact not the mechanism source. Next boundaries: signer-set governance, then mechanism-source content binding.
- Sprint 31: the Sprint-30 flat-key-list residual is closed — a public key is not permanent authority. `authorized_design_signers.json` is promoted to a governed registry (schema v0.2): each signer is `{public_key, scope, status, valid_from_tick, expires_at_tick, revoked_at_tick, rotated_to}`. `design_signing.verify_change_signature(change_set, registry, now_tick, change_scope)` proves cryptographic authorship FIRST (every Sprint-30 reason unchanged), then `signer_authority` decides whether the *genuine* signer is *currently* authorized for *this* change: `signer_revoked` (`status==revoked` or `now_tick >= revoked_at_tick`), `signer_expired` (`now_tick >= expires_at_tick`), `signer_not_yet_valid` (`now_tick < valid_from_tick`), `signer_wrong_scope` (the change's target control point not in `scope`, and `scope != ["*"]`). The crux is decision-time evaluation: the SAME genuine signature is authorized at tick 5 and `signer_revoked` at tick 20 for a signer revoked at tick 10 — authority is evaluated at decision time, not signing time, so a revoked key cannot replay a prior signature. A rotated successor (active, in window, in scope) is accepted while the revoked predecessor is rejected for new changes. Lifecycle is expressed in LOGICAL ticks (`evaluation_tick` per proposal, never wall-clock) and a release-gate asserts no wall-clock symbol appears in `design_signing`, so the gate stays reproducible. `design_authority` is preserved as active + wildcard scope so every Sprint 26–30 committed signature stays valid; the gate still only constrains a would-be ACCEPT and never overrides a trace block (a governed-but-valid signer on a weakening still blocks by trace). Only PUBLIC keys are committed; the six governance scenarios were signed by an authoring tool (`author_governed_signers.py`) whose generated private keys were discarded. Honest remaining residual (no safe-default claim): governance is single-signer (no threshold/multi-signer/m-of-n — a single active key still suffices), and the bound unit is still the policy artifact, not the mechanism source. Next boundaries: mechanism-source content binding (Sprint 32), then threshold governance if still needed.
- Sprint 32: the Sprint-29/30 "bound unit is the policy artifact, not the mechanism source" residual is closed — a signed policy is not enough if the gate code can be changed unsigned. New `mechanism_provenance.py` binds every enforcement-code file (`verifier_engine.py`, `mutation_gateway.py`, `retrieval_policy.py`, `raw_episode_store.py`, `trace_diff.py`, `change_provenance.py`, `design_signing.py`, `effect_classifier.py`, `project_self_audit.py`, and itself) by content hash in `mechanism_source_manifest.json`, keyed by role. Three layers: (1) **integrity** — `--verify` recomputes each file's hash from disk and fails release on any divergence; `project_self_audit.audit_project` reports `mechanism_source_binding` (verified/violated) and gates `strict_pass` on it, so a gate-code change without regenerating the manifest fails the strict audit too (two nets). (2) **provenance** — a `change_set` with `binding=="mechanism_source"` binds the literal before/after content of a real source file; `verify_mechanism_change_provenance` requires the pre-image to equal the CURRENT on-disk gate code (`stale_pre_image` otherwise) and binds the post-image + diff_digest; `trace_diff.derive_effect_from_trace` dispatches it to `_derive_mechanism_effect` instead of the policy path, and `combine_effects`' locked-block list gained the mechanism provenance failures. (3) **behavioral** — `probe_outcome_for_proposed_source` evaluates the bound gate's protected behavior against the PROPOSED post-image WITHOUT executing it: it parses the post-image, extracts the bound function (`adjudicate`), and safely INTERPRETS that function over the probe's fixed inputs across a restricted AST subset (if / boolean / comparison / return over parameters and literals); anything outside that subset — a call, attribute access, import, loop, module-level statement — is never evaluated and fails closed to `mechanism_probe_error` (a regression, which blocks). So a weakening of `adjudicate` is caught by probe even with clean policy files and a valid signature (`signature_status: signature_verified`, `effect_authority: trace_behavior_regression`), and the probe cannot read, write, or otherwise act. The unsigned mechanism-source change to a locked gate still blocks on the Sprint-30 signature gate. Only the `adjudicator` role is behaviorally probed; the other 9 roles are integrity-bound. Five scenarios authored by `author_mechanism_scenarios.py` (signed ones use a generated `mechanism_authority` key, discarded; only its public key committed). Two sabotage proofs: weakening the REAL `verifier_engine.py` on disk while leaving the policy clean fails release (manifest hash mismatch + probe-integrity), and making the probe always return the protected outcome fails release (the weakening scenarios wrongly accept) — each restored byte-identical. VERIFIER-FOUND DEFECT (fixed before close): the adversarial panel reproduced a real HIGH against the FIRST design, which ran the proposed post-image in a subprocess with the gate process's filesystem access — a probe-passing post-image could write files via module-level code during evaluation (restricted builtins would not fix this — the `().__class__.__subclasses__()` object-graph gadget escapes any in-process sandbox). The probe was reworked to the no-execution AST interpreter above, and selftest cases lock the exact attack (a module-level backdoor write creates no file; the proposed code is never run). Honest remaining residual (no safe-default claim): (1) only the adjudicator is behaviorally probed against a proposed change — the other roles' proposed changes fail closed to needs_review rather than being probe-tested; (2) the interpreter supports only a restricted AST subset, so a behavior-preserving adjudicator change using a construct outside it (a helper call, a loop) fails closed to a regression rather than accepting — a precision cost in the safe direction; (3) the manifest verifier `mechanism_provenance.py` binds itself but cannot fully attest itself (the who-watches-the-watchmen fixed point of a single-repo self-check; bounded by the behavioral probes + regression suite, not a watcher above it); (4) governance is still single-signer. Next boundaries: behavioral binding for more mechanism roles, then threshold/multi-signer governance if still needed.
- Sprint 25 (prior entry): that residual is closed. `effect` is no longer read from the proposal as authority. `effect_classifier.derive_effect` computes a deterministic lexical semantic diff (permissive verbs on a protected object the invariant guards → weaken/contradict; preservation markers without a permissive verb → extend/preserve; otherwise → needs_review) of the proposal claim vs. the invariant claim. The derived effect drives the design verifier rules; any declared `effect` is kept only as an untrusted hint and a declared/derived family mismatch is surfaced as `effect_mislabel`. A weakening mislabelled `extend` is reclassified and blocked; a weakening with no declared effect is still blocked from evidence; a weakening phrased without a permissive verb (e.g. "becomes advisory rather than mandatory") is caught by a weakening-phrase lexicon; and touching a protected object without preservation evidence lands in `needs_review`, which blocks rather than auto-accepting. The runtime `adjudicate` still confirms the hard-contradiction → `reject_episode` outcome on the derived effect. An adversarial verifier initially found that the first cut routed a protected-object-touch with no permissive verb to `preserve`/accept while the docs claimed it blocked; that was fixed (rule defaults to `needs_review`) and locked with `design_effect_lexicon_avoiding_weaken` + `design_effect_ambiguous_needs_review`. Honest remaining residual (no safe-default claim): a claim pairing an explicit preservation marker with a weakening phrased entirely outside both the permissive-verb and weakening-phrase lexicons can still reach accept; the next boundary is a trace-grounded diff against the invariant's regression scenarios — detect a weakening by what it would break, not by the words it uses.
